# 🚀 Deployment Guide

How to deploy **ProjectManager** for free: **Frontend → Vercel**, **Backend → Render**,
**Database → MongoDB Atlas**, **Files → Cloudinary**.

```
┌──────────────┐      HTTPS REST       ┌──────────────┐      ┌────────────────┐
│   Vercel     │  ───────────────────► │    Render    │ ───► │ MongoDB Atlas  │
│ (frontend)   │   VITE_API_BASE_URL   │  (backend)   │      └────────────────┘
└──────────────┘                       │  + Cloudinary│ ───► Cloudinary / Pusher / Google
                                       └──────────────┘
```

---

## 0. Prerequisites (create free accounts)
- [MongoDB Atlas](https://cloud.mongodb.com) — `MONGO_URI`
- [Cloudinary](https://cloudinary.com) — file uploads *(optional)*
- [Pusher](https://pusher.com) — real-time *(optional)*
- [Google Cloud Console](https://console.cloud.google.com) — OAuth + Drive *(optional)*
- A [GitHub](https://github.com) repo (done ✅) + [Vercel](https://vercel.com) + [Render](https://render.com) accounts

---

## 1. Database — MongoDB Atlas
1. Create a free **M0** cluster.
2. **Database Access** → add a user + password.
3. **Network Access** → allow `0.0.0.0/0` (fine for a demo).
4. **Connect → Drivers** → copy the SRV string → this is your `MONGO_URI`.

---

## 2. Backend — Render
**Option A — Blueprint (recommended):**
1. Render dashboard → **New + → Blueprint** → select this repo.
2. Render reads [`render.yaml`](render.yaml) and creates the `projectmanager-api` service.
3. Paste the secret env vars when prompted (at minimum `MONGO_URI`).
4. Deploy → note the URL, e.g. `https://projectmanager-api.onrender.com`.

**Option B — Manual:**
1. **New + → Web Service** → connect repo → **Root Directory:** `backend`.
2. Build: `npm install` · Start: `npm start` · Health check path: `/health`.
3. Add env vars (see checklist below) → Deploy.

**Seed the database** (one-time): Render **Shell** tab → `npm run seed`
(or run locally with the production `MONGO_URI`).

---

## 3. Frontend — Vercel
1. Vercel → **Add New → Project** → import this repo.
2. **Root Directory:** `frontend` (Vite is auto-detected; [`vercel.json`](frontend/vercel.json) handles SPA routing).
3. **Environment Variables:**
   - `VITE_API_BASE_URL` = your Render backend URL (no trailing slash)
   - *(optional)* `VITE_PUSHER_KEY`, `VITE_PUSHER_CLUSTER`, `VITE_GOOGLE_CLIENT_ID`, `VITE_GOOGLE_API_KEY`
4. Deploy → note the URL, e.g. `https://projectmanager.vercel.app`.

---

## 4. Wire them together (important!)
Back in **Render**, set/update:
- `FRONTEND_URL` = your Vercel URL
- `BACKEND_URL` = your Render URL
- `CORS_ORIGINS` = your Vercel URL

If using **Google OAuth**, add the callback to Google Cloud → Credentials → Authorized redirect URIs:
`https://<your-render-url>/api/auth/google/callback`

Redeploy the backend so the new env vars take effect.

---

## 5. Smoke test
1. Open the Vercel URL → log in with the seeded demo admin (`admin@demo.com` / `Demo@12345`).
2. Create a project/task, upload a file, view the dashboard + dependency graph.
3. `GET https://<render-url>/health` should return `{ "status": "ok", "db": "connected" }`.

---

## ⚙️ Environment variable checklist

| Variable | Where | Required? |
|---|---|---|
| `MONGO_URI` | Render | ✅ Yes |
| `JWT_SECRET` | Render | ✅ Yes (auto-generated by blueprint) |
| `FRONTEND_URL`, `BACKEND_URL`, `CORS_ORIGINS` | Render | ✅ after frontend deploy |
| `VITE_API_BASE_URL` | Vercel | ✅ Yes |
| `CLOUDINARY_*` | Render | ⚪ for file uploads |
| `PUSHER_*` / `VITE_PUSHER_*` | Render / Vercel | ⚪ for real-time |
| `GOOGLE_*` / `VITE_GOOGLE_*` | Render / Vercel | ⚪ for Google login/calendar |
| `EMAIL_USERS`, `EMAIL_PASSWORD` | Render | ⚪ for email |

> Without the optional ones, those features degrade gracefully — the app still runs.

## ⏰ Note on Render's free tier (cold starts)
The free backend **sleeps after ~15 min idle**, so the first request after a nap takes
**30–60s**. This repo includes a **keep-alive** solution:

### ✅ Included: GitHub Actions keep-alive
[`.github/workflows/keepalive.yml`](.github/workflows/keepalive.yml) pings `/health` every 12 min.
**To enable it:** GitHub repo → **Settings → Secrets and variables → Actions → Variables** →
add a variable `BACKEND_HEALTH_URL = https://<your-render-url>/health`.
(Until set, the workflow safely no-ops. You can also trigger it manually from the Actions tab.)

> Note: GitHub disables scheduled workflows after 60 days of repo inactivity — a periodic commit
> (or the manual "Run workflow" button) re-enables it.

### Alternatives
- **UptimeRobot / cron-job.org** — a free external monitor hitting `/health` every ~12 min (most
  reliable; also gives you uptime stats).
- **Upgrade** Render to a paid instance (~$7/mo) for always-on, no pinging needed.
